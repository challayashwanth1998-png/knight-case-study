1 Document Classification
Approach: Content-based classification using Google Gemini AI. All documents in a submission are batched into a single API call for efficiency.
  • Never uses filenames — filenames are often misleading (e.g., "Drivers.xlsx" might contain vehicle data). The classifier analyzes actual document content only.
  • Supports 11 document types: insurance application, driver list, equipment list, IFTA report, loss run, MVR, driver’s license, certificate of insurance, financial document, DOT safety record, and “other.”
  • Each classification includes a confidence score (0–100) and reasoning so underwriters can verify why a document was categorized.
  • Batching all documents in one call reduces API costs and latency vs. classifying each individually.
⚠ Assumption: Filenames are intentionally unreliable (per guidelines). The system treats filenames as opaque identifiers and never uses them for classification or routing decisions.
2 Data Extraction Strategy
Approach: Hybrid extraction — Python regex + structured parsing for machine-readable docs, Gemini AI only where needed.
  • Excel/CSV files → Parsed with openpyxl / pandas + regex patterns. Zero AI cost.
  • PDF text → Extracted with PyMuPDF (fitz), then structured with regex. Zero AI cost for text-based PDFs.
  • CDL images → Gemini Vision OCR. Only images require AI calls for extraction.
  • Each document type has a specific extraction schema (e.g., insurance applications extract 50+ fields including business info, coverage, safety features).
  • Deduplication: Cross-source name matching merges driver records from multiple documents (driver list + CDL photos + MVRs) into unified records.
⚠ Assumption: Cost efficiency matters. We minimize AI API calls by using free local extraction (regex, openpyxl) for structured formats and reserving AI for unstructured content (images, ambiguous text).
3 Handling Poor-Quality Documents
Approach: Graceful degradation with fallback chains and explicit null handling.
  • Missing text: If PyMuPDF extracts empty text from a PDF (scanned document), the system falls back to Gemini Vision OCR using the raw file bytes.
  • Partial data: Extraction schemas use null for any missing values. The pipeline continues even if key fields are missing — the rules engine flags gaps as warnings.
  • Corrupted files: Each file type processor wraps extraction in try/catch. A corrupt Excel file won’t crash the pipeline; the document is logged as “extraction_failed” and processing continues.
  • Low confidence: If classification confidence is below threshold, the document is still processed but flagged in the underwriter view for manual review.
  • Image quality: CDL images processed through ThreadPoolExecutor in parallel. If Gemini Vision can’t read a license, default fields are set to null and rules flag missing CDL data.
⚠ Assumption: Real-world submissions will have incomplete data. The system is designed to produce the best possible assessment with whatever data is available, rather than rejecting submissions outright for missing documents.
4 Identifying Conflicting Information
Approach: AI-powered cross-document conflict detection as a dedicated pipeline step.
  • After data extraction, all unified data from every document is passed to a dedicated Gemini conflict analysis call.
  • The AI compares values across documents: driver counts on the application vs. actual driver list, vehicle counts vs. equipment schedule, stated addresses, etc.
  • Each conflict includes: severity (critical/high/medium/low), type, description, source documents, and conflicting values found.
  • Critical conflicts (e.g., 10 drivers listed but only 6 CDLs provided) directly impact the risk score and can trigger a REFER decision.
  • Conflicts are displayed prominently in the underwriter view with color-coded severity indicators.
⚠ Assumption: Inconsistencies between documents are common and expected in insurance submissions. Some are benign (minor name spelling differences), others are critical (missing CDLs). The system surfaces all of them and lets the rules engine decide severity.
5 How AI Recommendations Are Generated
Approach: 4 parallel Gemini API calls for comprehensive risk assessment, followed by a recommendation synthesis step.
  • Company Risk: Evaluates business profile, DOT/MC number validity, years in operation, operation type, radius of operation.
  • Driver Risk: CDL status, violation history, age analysis, experience years, driver-to-vehicle ratio.
  • Fleet Analysis: Vehicle age distribution, VIN validation, fleet composition, stated values, maintenance indicators.
  • Financial & Loss: Claims history, loss ratios, open vs. closed claims, mileage patterns, premium adequacy.
  • All 4 analyses run in parallel using ThreadPoolExecutor to minimize latency (~15s instead of ~60s sequential).
  • Results include a composite risk score, narrative summary, and specific recommendations (premium adjustments, coverage conditions, exclusions).
⚠ Assumption: AI analysis is advisory only. The system generates recommendations but final underwriting decisions should always involve human review. The REFER decision exists specifically for edge cases where AI confidence is insufficient.
6 How Business Rules Are Applied
Approach: 21 configurable rules organized in 5 categories, evaluated against extracted + AI-analyzed data.
  • Rule categories: Eligibility (4), Driver (5), Vehicle (4), Safety (4), Financial (4).
  • Each rule produces a result: PASS, FAIL, WARNING, or INFO with configurable severity.
  • Rules are pure Python functions — no AI calls, instant evaluation. A deterministic layer on top of probabilistic AI analysis.
  • Rules are configurable at runtime: enable/disable individual rules, change severity levels, adjust thresholds via API or settings UI.
  • Decision logic: Any FAIL → DECLINE. Any WARNING (no FAILs) → REFER. All PASS → ACCEPT.
  • Rules with missing data default to WARNING (not FAIL) because the case study notes some documents are intentionally omitted.
⚠ Assumption: Some required documents may not be provided in every submission (as stated in the guidelines). Rules handle missing data gracefully — absent driver licenses trigger a WARNING, not an automatic DECLINE, allowing underwriter discretion.
7 How Results Are Presented to Users
Approach: Multi-view underwriter interface with progressive disclosure of details.
  • Dashboard: KPIs at a glance — total submissions, decision distribution (accept/refer/decline), AI cost tracking, recent submission feed.
  • Submission Detail: Tabbed view with Summary (decision + risk), Documents (classified files), Rules (21 rule results with pass/fail), Raw Analysis (AI output).
  • Real-time Pipeline: 6-step progress indicator shows extraction → classification → extraction → analysis → rules → decision in real time.
  • Compare View: Side-by-side comparison of two submissions with metrics diff — useful for renewal assessments.
  • Analytics: Token usage trends, processing time charts, rule pass rates, cost per submission over time.
  • Health Monitor: DB latency, AI API status, disk usage, cumulative costs — operational visibility.
📌
Key Assumptions Summary
  • Filenames are unreliable — Classification is content-based only (per case study guidelines).
  • Documents may be missing — The system processes whatever is provided and flags gaps as warnings, not failures.
  • Data inconsistencies are expected — Cross-document conflicts are detected and surfaced, not silently ignored.
  • Cost efficiency is important — Regex/Python extraction for structured docs (free), AI only where needed (~$0.005/submission).
  • AI is advisory — Risk analysis generates recommendations; final decisions involve human review via REFER pathway.
  • Processing speed matters — Parallel AI calls + thread pooling bring pipeline time from ~4 min to ~53 seconds.
  • Rules should be configurable — Business rules can be enabled/disabled and severity adjusted without code changes.
  • Document types may expand — The classifier supports an “other” category for unrecognized document types.
  • Audit trail is mandatory — Every action (upload, process, view, delete) is logged with timestamp and metadata for compliance.
  • Scalability from day one — Dockerized, AWS-ready, with S3/CloudWatch integration ready for production scale.
